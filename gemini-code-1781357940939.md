# 報告書 兼 開発手順書：Workspace Studioを活用したプロジェクト情報集約・管理自動化の検討

## 1. はじめに（背景と目的）

本プロジェクトは、社内開発案件における情報管理の効率化と「情報の属人化」「確認漏れ」の撲滅を目的とする。
現状、プロジェクトの関連情報は「Google Meet（会議）」「Slack（チャット）」「Confluence（設計資料）」に分散しており、これらを人間が手動で集約・管理するには多大な工数がかかっている。

本報告では、これらの「非構造化データ」を**Google Workspace Studio**および**Gemini**を活用してノーコードで1箇所に集約し、自動的に構造化されたタスク・課題管理を行うための全体構成および具体的な構築手順を定義する。

---

## 2. 全体システム構成案

システムは、原本データを格納する**「知識リソース層（Google ドライブ）」**と、AIが抽出したタスクを管理する**「タスク管理層（Google スプレッドシート）」**の2層で一箇所に集約・管理する。

【各データソース】 【自動化（Workspace Studio）】 【一箇所に集約（保管先）】
├─ Google Meet (文字起こし) ──▶ (自動保存 / 決定事項抽出) ──────▶ 📂 Google ドライブ専用フォルダ
├─ Slack 会話 (1日2回バッチ) ─▶ (非構造化 ➔ 構造化変換) ─────▶ ├─ 原本ドキュメント
├─ Confluence (設計資料) ───▶ (仕様変更の差分抽出) ─────▶ └─ 専門用語集シート
└─ その他関連資料 (スケジュール) ───────────────────────┘ │
▼
📊 タスク管理スプレッドシート
(進捗・ステータスの自動更新)

### 2.1. データ保管先（器）の設計

情報を一箇所にまとめるため、Google Workspace上に以下の2つの保管先を事前に用意する。

1. **原本・知識の保管先：Google ドライブ（専用フォルダ）**
    - **役割**: すべての非構造化データ（Meet文字起こし、Confluenceのバックアップ、仕様書、スケジュール等）を格納する「信頼できる唯一の情報源（Single Source of Truth）」。
2. **タスク・進捗の保管先：Google スプレッドシート（タスク・課題管理シート）**
    - **役割**: 各ソースからGeminiが抽出したタスクを、行と列に整理して一元管理・自動更新する。

## 3. 事前準備：データ基盤の作成

### 手順3.1：Google ドライブのフォルダ構成作成

Google ドライブ内にプロジェクト専用のメインフォルダを作成し、配下に以下のサブフォルダを構築する。

- `[プロジェクト名] _管理フォルダ`
    - `01_Meet文字起こし`（※Meetの自動保存先リンク、またはショートカットを配置）
    - `02_Confluenceバックアップ`（※自動生成された仕様ドキュメントの格納先）
    - `03_プロジェクト定義・スケジュール`（※要件定義書やマイルストーンを格納）

### 手順3.2：タスク・課題管理スプレッドシートの作成

`[プロジェクト名]_タスク・課題管理シート` を作成し、最初のワークシートに以下の列（ヘッダー）を設定する。

- **A列**: 起票日
- **B列**: タスク・課題名
- **C列**: 担当者
- **D列**: 期限
- **E列**: ステータス（初期値：未着手）
- **F列**: 最新状況・メモ（会話の経緯や変更理由の要約）
- **G列**: 元データリンク（SlackメッセージURLやConfluenceのページURL）

### 手順3.3：人間とGeminiの「専門用語集シート」の作成

Google Meetの誤変換対策およびGeminiのコンテキスト理解度を向上させるため、上記スプレッドシートに「専門用語集」タブを追加し、以下の列を設定する。

- **A列**: 正しい表記（例：`Apex`、`LWC`、`Agentforce`）
- **B列**: よくある誤変換（例：`エーペックス`、`営業担当`、`エージェントフォース`）
- **C列**: 意味・説明
- **D列**: 最終更新者（`人間` または `Gemini`）
- **E列**: 確認ステータス（`確認済み` または `要確認`）

## 4. Workspace Studio 構築手順（逆引きマニュアル）

### フロー①：Slack会話内容の定期抽出・構造化（1日2回バッチ）

Slackの非構造化データから「課題」と「タスク」を1日2回自動抽出し、スプレッドシートに追記・差分更新するフロー。

1. **トリガー（Starter）の設定**
    - コンポーネントに「Timer（タイマー）」を選択。
    - スケジュールを「毎日（Every day）」に設定し、実行時間を `12:00` と `18:00` の2回指定する。
2. **Slackデータの取得（Action）**
    - コンポーネントに「Slack」を選択。
    - アクションに「Get messages（メッセージの取得）」を選択。
    - 対象チャンネル（例：`#project-dev`）を指定し、取得範囲を「過去12時間分」かつ「スレッドの返信（Replies）を含む」に設定する。
3. **現在の管理シートの読み込み（Action）**
    - 「Google スプレッドシート」コンポーネントを挟み、現在シートに登録されているタスク一覧（特にG列のSlackリンク）を読み込ませる。
4. **Geminiによる分析と構造化（Action）**
    - 「Gemini」コンポーネントを追加し、データソースに「ステップ2のSlackデータ」と「ステップ3の現シートデータ」を指定。
    - **プロンプト（指示文）**:

        ```text
        あなたは優秀なPMアシスタントです。渡されたSlackの会話履歴（スレッド含む）と現在の管理シートを比較分析してください。

        【処理ルール】
        1. 既存タスクの更新:
           Slackのスレッド内で、すでにシートに存在する課題（Slackリンクが一致するもの）の進捗やステータス（例：未着手➔完了）に変更、または新しい進捗メモがあれば、最新状態に更新するデータを作成してください。
        2. 新規タスクの追加:
           シートに存在しない新しい課題やタスクがSlack上で発生している場合、新規行として抽出してください。
        3. 構造化出力:
           必ず次の順番でタブ区切り（CSV形式）のテキストのみを出力してください。
           [起票日] \t [タスク名] \t [担当者] \t [期限] \t [ステータス] \t [最新状況・メモ] \t [Slackリンクまたは添付ファイルURL]
        ```
5. **スプレッドシートへの書き込み（Action）**
    - 「Google スプレッドシート」のアクション「Append row（行の追加）」または、既存行の更新ロジックを配置し、Geminiが出力した構造化データを流し込む。

### フロー②：Confluence仕様書の変更検知とタスク自動起票

Confluenceに記載された設計ドキュメントの更新を検知し、タスクを自動生成・バックアップするフロー。

1. **トリガー（Starter）の設定**
    - コンポーネントに「Confluence」を選択。
    - トリガー条件に「ページが作成または更新されたとき（When a page is created or updated）」を選択。
2. **Confluenceデータの取得（Action）**
    - 該当ページのテキスト内容および変更履歴、ページURLを取得。
3. **バックアップの保存（Action）**
    - 「Google ドライブ」コンポーネントを選択し、取得したテキストをGoogle ドキュメント形式に変換。
    - ドライブ内の `02_Confluenceバックアップ` フォルダへ自動保存（上書き）する。
4. **Geminiによるタスク抽出（Action）**
    - 「Gemini」コンポーネントを追加し、Confluenceのテキストを渡す。
    - 事前に作成した「専門用語集シート」を同時に参照ソースとして読み込ませる。
    - **プロンプト（指示文）**:
        ```text
        添付の専門用語集シートを辞書として使用し、Confluenceの変更内容を読み込んでください。
        この仕様変更に伴い、新しく発生する開発タスク（LWC修正、Apexクラス改修など）、担当者、期限を抽出してください。
        結果は管理シートのヘッダー形式に合わせて構造化データとして出力してください。URL列には、このConfluenceページのURLを記載してください。
        ```
5. **管理シートへの追記（Action）**
    - 抽出されたタスクデータを「タスク・課題管理シート」の最終行に追加する。

## 5. 運用のコツとAI精度の向上対策

### 5.1. 会話のノイズを減らす「Slack運用ルール」の策定

AIが雑談や挨拶をタスクとして誤認識するのを防ぐため、チーム内で以下の簡単な運用ルールを共有する。

- 課題を報告したり、タスクを依頼したりする際は、メッセージの文頭に `【課題】` または `【タスク】` と明記する。
- Workspace Studio側のGeminiプロンプトに、「文頭に【課題】や【タスク】とついているメッセージを最優先で抽出すること」という一言を付け加えることで、抽出精度が100%に近づく。

### 5.2. 添付ファイル・URLのハンドリング

- Slackにファイル（画像やPDF）が添付された場合、Workspace Studioはファイルそのものではなく、Slack上の「ファイルダウンロードURL」をテキストとしてキャッチする。
- Geminiのプロンプトで「ファイルURLがある場合は、それを『添付ファイル・URL』列に記載せよ」と指示しておくことで、シートからSlack上の原本ファイルへワンクリックでアクセスできる環境を維持する。

### 5.3. 専門用語集の共同メンテナンス

- プロジェクトの進行に伴い、新しい用語や、Meet文字起こしの新しい誤変換パターンを発見した場合、人間が「専門用語集シート」に追記する。
- また、フロー実行時にGemini側にも「新しい固有名詞や誤変換パターンがあれば、ステータス『要確認』・更新者『Gemini』として最終行に仮登録せよ」と指示しておく。
- 人間は週に1回程度シートをレビューし、Geminiの仮登録が正しければステータスを「確認済み」に変えるだけで、使えば使うほど文字起こし補正の精度が自動的に向上するエコシステムが完成する。
