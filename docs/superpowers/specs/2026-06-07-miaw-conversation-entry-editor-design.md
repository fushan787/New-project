# MIAW 会話エントリ編集 UI 設計

## 目的

Messaging In-App and Web (MIAW) の会話画面で、`MessagingSession` の詳細ページに配置した LWC から会話内容を表示し、会話エントリの編集・物理削除を API 経由で実行できるようにする。

この機能は、オンラインチャット運用中に混入した機微情報を後から安全に修正することを主目的とする。画面は Salesforce らしい見た目を保ちつつ、検証アプリとしても見やすい UI にする。

## 対象オブジェクト

- `MessagingSession`
- `Conversation`
- `ConversationEntry`
- カスタムオブジェクト `ログ`

## ゴール

- `MessagingSession` 詳細画面に LWC を配置する
- 会話を LINE 風のタイムラインとして表示する
- 各 `ConversationEntry` を編集・物理削除できる
- 編集画面はポップアップで表示する
- 編集画面内に編集前のメッセージを表示する
- 編集用テキストエリアには元の本文を初期値として入れる
- 保存時に確認メッセージを表示する
- 削除時にも確認メッセージを表示する
- 編集・削除の履歴として、誰が、いつ、何を操作したかだけを `ログ` に残す
- `ログ` には編集後の本文を保存しない
- 会話データの取得・更新・削除は Apex の DML ではなく Salesforce の会話 API を経由して行う
- Apex は UI と API のオーケストレーション、権限チェック、監査ログ記録を担う

## 非ゴール

- 会話本文の履歴保存や差分保存
- 監査ログの全文保持
- 非同期処理化
- 画面以外からの一括編集ツール
- 権限設計の大幅な再構築

## UI 方針

- Salesforce API 検証アプリだが、画面はきれいに作る
- Salesforce らしい見た目を優先する
- SLDS 2 をベースにする
- 強い独自デザインではなく、標準 Lightning UI に自然に馴染む構成にする
- 会話表示は 1 カラムの縦スクロールを基本とし、送信者ごとの差をバブルや配置で見分けやすくする
- 危険操作である削除は、色とラベルで明確に区別する

## 画面構成

### 1. ヘッダー領域

- `MessagingSession` の基本情報を表示する
- 例: セッション名、関連する Conversation、状態、最終更新日時
- 必要最小限の情報だけを表示し、会話本体を主役にする

### 2. 会話タイムライン

- `ConversationEntry` を時系列で表示する
- 各メッセージは送信者、送信日時、本文を見やすく表示する
- LINE 風の読みやすさを意識しつつ、SLDS ベースのカード表現を使う
- 各メッセージに `編集` と `削除` の操作を付ける

### 3. 編集モーダル

- `lightning-modal` を使う
- 編集前の本文を参照表示する
- 編集用テキストエリアには元の本文をデフォルトでセットする
- 保存前に確認ダイアログを出す
- 保存成功後はタイムラインを再取得し、表示を更新する

### 4. 削除確認

- 削除操作の前に確認ダイアログを出す
- 誤削除防止のため、本文の一部または送信日時を確認文面に含める
- 削除成功後はタイムラインを再取得する

## コンポーネント構成

### LWC: `conversationTimeline`

役割:

- `MessagingSession` に紐づく会話の取得
- タイムライン描画
- 編集モーダルの開閉
- 削除確認の実行
- Apex 経由で会話 API を呼び出すための画面起点
- API 処理後の再描画

依存:

- Apex クラス `ConversationEntryController`
- Salesforce 会話 API への callout
- `lightning-modal`
- `lightning-confirm` 相当の確認 UI か、同等の確認モーダル実装

### Apex: `ConversationEntryController`

役割:

- `MessagingSession` をキーに会話 API から会話データを取得する
- `ConversationEntry` の編集要求を受け付け、API に送る前の入力検証を行う
- `ConversationEntry` の物理削除要求を受け付け、API に送る前の入力検証を行う
- API のレスポンスを LWC 向け DTO に整形する
- 成功時に監査ログを書き込む

### Apex: `ConversationEntryAuditService`

役割:

- `ログ` レコードを生成する
- 編集・削除の監査記録を 1 箇所に集約する
- 監査対象に会話本文を含めない

## データフロー

1. ユーザーが `MessagingSession` 詳細画面を開く
2. LWC が現在の `MessagingSession` ID を受け取る
3. LWC が Apex に問い合わせて会話データ取得を依頼する
4. Apex が Salesforce の会話 API を呼び出して会話データを取得する
5. LWC はタイムラインとして描画する
6. ユーザーが `編集` を押す
7. モーダルが開き、元本文を表示し、編集欄に初期値として入れる
8. ユーザーが保存を押すと確認メッセージを表示する
9. 確定後、Apex が会話 API を呼び出して元のエントリを削除し、新しい内容で再作成する
10. 同一トランザクションで `ログ` を作成する
11. ユーザーが `削除` を押すと確認メッセージを表示する
12. 確定後、Apex が会話 API を呼び出して対象エントリを物理削除し、同一トランザクションで `ログ` を作成する

## 監査ログ設計

`ログ` には以下を残す。

- 操作種別: 編集 / 削除
- 操作者
- 実行日時
- 対象 `ConversationEntry` ID
- 対象 `MessagingSession` ID
- 操作対象の識別に必要な最小限の情報
- API 呼び出しの結果ステータス

`ログ` には以下を残さない。

- 編集後の本文
- 編集前の本文全文
- 会話全体の全文コピー
- 機微情報の再保存につながる詳細内容

本文そのものは記録せず、対象レコードの ID と操作種別で追跡する。

## 権限と制御

- 編集・削除は、対象レコードを参照できるユーザーに限定する
- Apex 側で権限チェックを行う
- `ConversationEntry` の更新可否と削除可否を分けて扱う
- 権限不足時は、UI に操作不可状態またはエラーを返す

## エラーハンドリング

- 取得失敗時はタイムライン領域にエラーを表示する
- 更新失敗時はトーストで通知し、編集モーダルは閉じない
- 削除失敗時はトーストで通知し、対象メッセージは保持する
- 監査ログ作成に失敗した場合は、会話更新/削除も失敗として扱う
- 失敗理由はユーザー向けには簡潔に、詳細はログに残す

## 実装方針

- 取得処理は Apex の `@AuraEnabled(cacheable=true)` を使い、内部で会話 API を呼ぶ
- 更新・削除は imperative 呼び出しで行い、Apex が会話 API を呼ぶ
- 更新・削除とログ作成は 1 回の Apex トランザクションにまとめる
- 会話 API 呼び出しは Named Credential または外部認証設定を使って実施する
- LWC の表示は標準コンポーネントを中心に組み立てる
- 画面の見た目は SLDS 2 のトークンとユーティリティクラスを優先する

## テスト方針

### Apex

- 会話 API の GET 呼び出し結果を DTO に正しく変換できること
- 編集時に「削除 + 再作成」の API フローが呼ばれること
- 削除時に削除 API が呼ばれること
- API エラー時にトランザクションがロールバックされること
- `ログ` には本文が入らないこと

### LWC

- タイムラインが正しい順序で表示されること
- 編集モーダルで元本文が初期値として入ること
- 保存・削除の前に確認ダイアログが出ること
- 成功時に再描画されること
- 失敗時にユーザーへ明確な通知が出ること

## 受け入れ条件

- `MessagingSession` 詳細画面に LWC を配置できる
- 会話が見やすいタイムラインで表示される
- 各 `ConversationEntry` を編集できる
- 各 `ConversationEntry` を物理削除できる
- 編集/削除前に確認が入る
- 編集・削除の履歴が `ログ` に残る
- 編集後の本文は `ログ` に保存されない
- 画面の見た目が Salesforce 標準 UI と大きく乖離しない
- 取得・編集・削除の実体は Apex から会話 API 経由で実行される

## 実装前の確認事項

- `ログ` オブジェクトの項目 API 名
- 使用する会話 API が Conversation Data API か Connect REST API か
- 会話 API の認証方式と Named Credential の要否
- `ConversationEntry` の編集フローが「削除 + 再作成」で成立するか
- API の削除対象が個別メッセージまで対応しているか
