# Salesforce API連携 Python CLI 検討メモ

## 前提

新しいPythonプロジェクトとして、手元でコマンド実行するSalesforce API連携CLIツールを作る。

想定する主な機能は以下。

- Salesforceデータの検索
- Salesforceデータの更新
- Salesforceレポートのエクスポート

まずは認証方式として、以下の2案を比較した。

- ユーザー名 + パスワード + セキュリティトークン
- OAuth 2.0 / Connected App

## 認証方式の比較

### A. ユーザー名 + パスワード + セキュリティトークン

#### メリット

- 実装がシンプル。
- CLIや検証用スクリプトで早く動作確認できる。
- Salesforce側のConnected App設定が最小限で済む場合がある。
- `simple-salesforce` などのPythonライブラリと相性がよい。

#### デメリット

- Pythonプログラム側でSalesforceのパスワードとセキュリティトークンを扱う必要がある。
- パスワード変更、セキュリティトークン再発行、MFA、ログインポリシー変更の影響を受けやすい。
- 個人ユーザーの資格情報に依存すると、退職・異動・権限変更で運用が壊れやすい。
- 本番連携としてはセキュリティ面で弱い。
- IP制限や組織設定によって接続できない場合がある。

#### 向いているケース

- Salesforce API接続の初期検証。
- 自分だけが使う社内CLI。
- Sandboxや開発環境での検証。
- 短期間の移行・抽出ツール。

### B. OAuth 2.0 / Connected App

#### メリット

- Salesforce API連携として本番運用に向いている。
- パスワードをCLIに保存しない構成にできる。
- アクセストークンやリフレッシュトークンで継続利用できる。
- Connected App側でスコープ、許可ユーザー、IP制限、トークンポリシーを管理しやすい。
- 監査・権限管理・複数ユーザー利用に向いている。
- 将来Webアプリ化、定期実行、他システム連携に広げやすい。

#### デメリット

- 初期設定がやや複雑。
- Salesforce側でConnected Appの作成、Callback URL、OAuth Scope設定が必要。
- CLIで初回認可フローをどう扱うか設計が必要。
- リフレッシュトークンの保存、更新、失効時の再認証処理が必要。
- 組織のセキュリティ設定によって管理者承認が必要になる場合がある。

#### 向いているケース

- 継続的に使うCLI。
- 本番Salesforceへの接続。
- 複数人で使う可能性があるツール。
- データ更新を含む連携。
- 監査や権限管理を重視する運用。

## 推奨方針

今回のCLIは、検索・更新・レポートエクスポートまで含む可能性があるため、最終的にはOAuth 2.0 / Connected Appを本番向けの標準方式にするのが望ましい。

ただし、開発初期はユーザー名 + パスワード + セキュリティトークン方式の方が早く検証できる。

そのため、以下の方針が現実的。

1. 認証処理を独立モジュールにする。
2. 最初はユーザー名 + パスワード + セキュリティトークンで動作確認する。
3. CLIの形が固まったらOAuth 2.0に切り替えられるようにする。
4. 本番利用ではOAuth 2.0 / Connected Appを標準にする。

つまり、ユーザー名 + パスワード + セキュリティトークン方式は開発用の近道、OAuth 2.0 / Connected Appは本番用の標準ルートとして考える。

## セキュリティトークンが必要な理由

Salesforceのセキュリティトークンは、ユーザー名・パスワードで外部アプリからログインするときの追加認証要素。

主な目的は、信頼済みIP範囲の外からAPIログインする場合に、パスワードだけで接続できないようにすること。

例えば、自宅PC、社外ネットワーク、クラウド環境など、Salesforce組織側で許可済みではないIPから接続する場合、通常は以下のように認証する。

```text
username = user@example.com
password = Salesforceパスワード + セキュリティトークン
```

Pythonコード上では、実質的に以下の形になる。

```python
password = "your_password" + "your_security_token"
```

セキュリティトークンがある理由は以下。

- パスワードだけを盗まれてもAPIログインされにくくする。
- 信頼済みIP範囲の外からのログインを追加チェックする。
- 外部ツール/API連携と通常のブラウザログインを区別しやすくする。

ただし、Salesforce管理者がログインIP範囲やネットワークアクセスの信頼済みIP範囲を設定している場合、セキュリティトークンが不要になることもある。

実務上の整理は以下。

- 社外・自宅・クラウドからPython CLIを使う場合、トークンが必要になりやすい。
- 社内固定IPから使う場合、不要な場合がある。
- OAuth / Connected Appを使う場合、セキュリティトークンではなくOAuthトークンで認証する。

## ユーザー名 + パスワード + セキュリティトークン方式のセキュリティリスク

この方式は手軽だが、本番運用ではリスクが高い。

### 資格情報が強すぎる

Salesforceの実ユーザーのパスワードを外部プログラムに持たせることになる。

漏えいすると、APIだけでなく通常ログインにも悪用される可能性がある。

### 保存場所の管理が難しい

`.env`、設定ファイル、端末の環境変数などに保存する必要がある。

誤ってGitにコミットすると危険。

ログやエラー出力にパスワードやトークンを出さない配慮も必要。

### MFAやログインポリシーと相性が悪いことがある

Salesforce組織の設定によっては、ユーザー名・パスワード方式が制限される場合がある。

IP制限、MFA、セッション設定、プロファイル権限の影響を受けやすい。

### 個人ユーザーに紐づけると運用が壊れやすい

その人が退職、異動、パスワード変更、権限変更をするとCLIが動かなくなる。

また、誰の権限で更新・取得されたのかが個人依存になる。

### 権限が広すぎると被害が大きい

更新機能を作る場合、誤操作や資格情報漏えいによって大量データ変更の危険がある。

APIユーザーに必要以上のオブジェクト権限や項目権限を与えるべきではない。

### トークン更新・失効時の管理が必要

セキュリティトークンをリセットすると既存CLIは接続不能になる。

パスワード変更でも再設定が必要になることがある。

## 検索とレポートエクスポートのみの場合のデータリスク

データ更新をしない場合、破壊的な更新事故のリスクは下がる。

ただし、Salesforceデータの情報漏えい、過剰取得、持ち出しリスクは残る。

### 閲覧権限の範囲内で大量データを取得できる

APIは認証ユーザーの権限で動く。

そのユーザーに広い権限、例えば `View All Data` があると、非常に広範囲のデータを取得できる可能性がある。

### CSV/Excelに出した後はSalesforceの制御外になる

Salesforce内では共有設定、項目レベルセキュリティ、監査ログなどで守られている。

しかし、一度ローカルPCにCSV/Excelとして保存すると、メール添付、クラウド同期、誤送信、端末紛失などのリスクが出る。

### 個人情報・機密項目を意図せず含める

取引先責任者、商談、ケース、活動履歴などには個人情報や営業機密が入りやすい。

検索だけのつもりでも、SOQLで項目を広く指定すると不要な項目まで取得する可能性がある。

### レポートエクスポート権限が強い

Salesforceでは、レポートを画面で閲覧することとファイルとしてエクスポートすることはリスクが異なる。

ファイルとして持ち出せる方が、情報漏えいリスクは高い。

### 認証情報の漏えい

ユーザー名 + パスワード + セキュリティトークン方式では、資格情報の保存が必要。

これが漏れると、そのユーザーの権限範囲でデータ検索・エクスポートされる可能性がある。

### 監査・追跡が弱くなる可能性

Salesforce側にはAPIログやログイン履歴を残せるが、ローカルに保存した後のファイル操作までは通常追跡できない。

誰が、いつ、どの条件で、どのファイルに出したかをCLI側でも記録した方が安全。

## 読み取り専用の共通Salesforce連携ユーザーを複数人で使う場合のリスク

読み取り専用のSalesforce連携ユーザーを作った場合でも、そのユーザーを複数人で共通利用すると別のリスクが発生する。

更新権限がないため破壊的なデータ変更リスクは抑えられるが、監査、責任分界、情報漏えい、権限管理の面では注意が必要。

### 誰が実行したか分からない

Salesforceのログ上は、全員が同じ連携ユーザーとして記録される。

そのため、誰が、いつ、どのデータを検索・エクスポートしたかを個人単位で追跡しにくい。

### 資格情報が広く共有される

ユーザー名、パスワード、セキュリティトークンを複数人に配ることになる。

退職者や異動者からの回収、漏えい時の影響範囲特定、再配布の管理が難しくなる。

### 1人の漏えいが全員の問題になる

誰か1人のPC、`.env`、メモ、チャット履歴などから資格情報が漏れると、共通ユーザーの権限でSalesforceデータを取得できる。

共通ユーザーを使っている場合、漏えい元や実行者の特定も難しくなる。

### 権限が最大公約数になりやすい

複数人が使うため、全員が必要とするデータを見られる権限に寄せがちになる。

その結果、本来は一部の人だけが見ればよいデータまで、共通ユーザー経由で全員が取得できる可能性がある。

### レポート/CSVの持ち出し責任が曖昧になる

Salesforce側では共通ユーザーによる操作として見えるため、誰がどのレポートやCSVを出力したかが分かりにくい。

個人情報や営業機密を含むエクスポートでは、監査対応やインシデント調査が難しくなる。

### パスワード変更やトークンリセットの影響が全員に出る

共通ユーザーのパスワード変更やセキュリティトークンリセットを行うと、全利用者のCLIが同時に接続できなくなる。

運用連絡、設定変更、再配布が必要になり、影響範囲が大きい。

### 利用者ごとのアクセス制御ができない

Aさんには取引先だけ、Bさんには商談も許可する、といった個人単位の制御がしづらい。

Salesforceの共有設定、項目レベルセキュリティ、監査ログを個人単位で活かしにくくなる。

## 共通連携ユーザーを使う場合の追加対策

複数人が使うCLIであれば、可能な限り共通ユーザーではなく、OAuth / Connected Appで各ユーザー自身に認可させる方式が望ましい。

共通ユーザーを使う場合は、以下の対策を追加する。

### 可能なら各ユーザー自身のSalesforce認証を使う

OAuth / Connected Appで個人ごとに認可する。

この方式ならSalesforce側の監査ログも個人単位で残り、個人ごとの権限や共有設定も反映しやすい。

### CLI側で利用者を記録する

共通ユーザーを使う場合でも、CLI側で以下を記録する。

- OSログインユーザー名
- 実行者メールアドレス
- 実行日時
- 対象オブジェクト
- SOQLまたは検索条件
- 取得件数
- 出力ファイル名

ただし、これはSalesforce側の個人単位監査の代替ではなく、補助的な記録として扱う。

### 共通ユーザーの権限を強く絞る

共通ユーザーには必要最小限の読み取り権限だけを付与する。

- 読み取り専用にする。
- 必要なオブジェクトだけ許可する。
- 必要な項目だけ許可する。
- `View All Data` は避ける。
- レポートエクスポート権限も必要最小限にする。

### 資格情報を直接配らない

`.env` や設定ファイルをメール、チャット、共有フォルダで配布しない。

可能であれば、社内のシークレット管理、OS資格情報ストア、端末ごとの安全な設定方式を使う。

### 利用者変更時の運用を決める

退職・異動・担当変更があった場合の運用を事前に決めておく。

- 利用者一覧を管理する。
- 退職・異動時はセキュリティトークンを再発行する。
- 定期的にパスワードやトークンをローテーションする。
- 連携ユーザーの利用目的と利用者を棚卸しする。

### 共通ユーザーが向いているケースを限定する

共通ユーザーは、自動バッチや限定された管理者だけが使うCLIには向いている。

一方で、一般利用者が複数人で使うCLIには向いていない。

複数人が日常的に使う場合は、OAuth / Connected Appによる個人認証を優先する。

## 推奨する安全対策

### 認証・ユーザー管理

- 専用のSalesforce連携ユーザーを作る。
- 個人ユーザーの資格情報を使わない。
- 連携ユーザーには必要最小限のプロファイル/権限セットだけ付与する。
- `View All Data` や `Modify All Data` は原則付けない。
- 本番用とSandbox用の資格情報を分ける。
- 複数人でCLIを使う場合は、共通ユーザーよりもOAuth / Connected Appによる個人認証を優先する。

### 資格情報の保存

- `.env` は `.gitignore` に入れる。
- パスワードやセキュリティトークンをログに出さない。
- 可能であればOSの資格情報ストアを使う。
- 設定ファイルに資格情報を直書きしない。

### 検索・エクスポート制御

- SOQLは必要項目だけを明示的に指定する。
- `SELECT *` 的な発想を避ける。
- 出力対象のオブジェクトを許可リストで制限する。
- 出力項目も許可リストで制限する。
- 機密項目は除外、またはマスキングする。
- CLIに `--limit` を設ける。
- 大量エクスポート時は確認プロンプトを出す。
- 出力先フォルダを固定し、Git管理外にする。

### 更新処理を含める場合

- 更新コマンドには `--dry-run` を用意する。
- 更新前に対象件数と条件を表示する。
- 一定件数以上の更新には明示的な確認を求める。
- 更新内容を監査ログに残す。
- 可能ならバックアップや事前エクスポートを行う。

### 監査

- 実行日時、実行ユーザー、対象オブジェクト、検索条件、取得件数、出力ファイル名をCLI側で記録する。
- エラー時にも資格情報をログに出さない。
- Salesforce側のログイン履歴やAPI利用状況も確認できるようにする。

## まとめ

ユーザー名 + パスワード + セキュリティトークン方式は、開発初期や検証では便利。

ただし、本番運用や継続利用では、資格情報の扱い、権限の広さ、データ持ち出し、監査の弱さが問題になりやすい。

検索とレポートエクスポートだけで更新しない場合でも、Salesforceデータをローカルファイルとして持ち出す時点で情報漏えいリスクが発生する。

また、読み取り専用の共通Salesforce連携ユーザーを複数人で使う場合、誰が実行したか分からない、資格情報が広く共有される、権限が最大公約数になりやすい、といったリスクがある。

したがって、CLIを作る場合は以下の設計が望ましい。

1. 開発初期はユーザー名 + パスワード + セキュリティトークン方式で接続確認する。
2. 認証処理は独立させ、OAuth 2.0 / Connected Appへ差し替え可能にする。
3. Salesforce連携ユーザーは読み取り専用または必要最小権限にする。
4. 検索・エクスポート対象を許可リストで制限する。
5. 出力ファイルと実行履歴を管理し、資格情報は安全に保存する。
6. 複数人で使うCLIでは、可能な限り個人認証を使い、共通ユーザー利用時はCLI側の実行ログと資格情報管理を強化する。
